When asking questions …
• Search before you ask.
• Be civil and professional. Write as if your coworkers/supervisor are reading
it (because in a way they are).
• Help us help you by being specific.
• Not useful: “My code doesn’t work. Here is a dump of my code and 50 lines of
backtrace. Help me!” No one but you should be determining what to type.
• Useful: “I specifically expected the below code to do X, but instead it does Y. I tried
searching for the error message, checking StackOverflow, etc. but I found nothing
relevant. The first place in the backtrace that my code is implicated is [line N]. Any
idea what I’m doing wrong?”

--- Page 25 ---
26
Extreme Programming (XP)
• If short iterations with customer are good, make them as short as
possible (weeks vs. years)
• If simplicity is good, always do the simplest thing that could possibly
work
• If testing is good, test all the time. In fact, write the test before you write
the code, then write the code to pass the test.
• If code reviews are good, review code continuously: program in pairs,
taking turns looking over each other’s shoulders.
• and turn every proposed code change into a mini-code-review (pull request)

Clusters & Warehouse-scale computers
• late 1990s: supercomputers ➔ late 2000s: datacenters of 100K’s of
commodity PCs (“warehouse scale”)
• Economies of scale in hardware and operations pushed down cost of largest
datacenters by factors 3X to 8X
Eliminates capital costs
• Scalability: to get more capacity, add more PCs
• Dependability: OK if a few are broken at any given time, if software architecture
supports it for SaaS startups
• Utility computing/public cloud computing (Amazon AWS, Google Cloud,
Microsoft Azure…)
• Traditional private datacenters were utilized 10% - 20%, so consolidate many 100s of
customers into a few warehouse-scale datacenters
• illusion of infinite scalability to multiple customers (“public cloud”):
1000 computers @ 1 hour cost same as 1 computer @ 1000 hours
